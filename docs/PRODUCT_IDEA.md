## 1. Product Vision

CodeFlow는 알고리즘 문제를 **"정답 맞히기"에서 끝내지 않는다.**

코드를 실행하고 → 그 과정이 어떻게 흘러가는지 눈으로 보고 → AI가 다음 학습 단계까지 연결해 주는 **인터랙티브 학습 플랫폼**을 지향한다.

| 핵심 가치 | 의미 |
|---|---|
| **Process First** | 결과보다 과정을 이해하게 하는 학습 경험 |
| **Personalized Flow** | 내 약점에 맞춰 다음 문제를 이어주는 학습 루프 |
| **Visual + AI** | 코드 실행 과정과 개념 설명이 분리되지 않는 제품 |

> [!IMPORTANT]
> CodeFlow의 핵심은 **"많이 푸는 것"이 아니라 "왜 이렇게 동작하는지 이해하는 것"**이다.

## 2. Why Now

기존 알고리즘 학습은 두 방향에 치우쳐 있다.

- 문제를 **많이 푸는** 방식
- 정답 코드를 **외우는** 방식

하지만 초보자에게 더 중요한 것은 **"왜 이 코드가 여기서 이렇게 움직이는가"** 를 이해하는 일이다.

CodeFlow는 바로 그 지점을 겨냥한다. 아래를 모두 눈으로 보여주고, AI가 그 과정을 설명하면서 학습자의 사고를 다음 문제로 이어준다.

| 시각화 대상 | 설명 |
|---|---|
| `Current Line` | 현재 실행 중인 코드 라인 |
| `Stack / Heap` | 메모리 상태 변화 |
| `Loop / Branch` | 반복·분기 흐름 |
| `Array / Pointer` | 인덱스와 포인터 이동 |

## 3. Core Experience

우리가 만들고 싶은 학습 흐름은 아래와 같다.

```
문제를 받는다
  ↓
코드를 직접 작성한다
  ↓
실행 과정을 시각화로 본다
  ↓
AI 설명으로 "왜 이렇게 동작하는지" 이해한다
  ↓
같은 유형의 다음 문제를 추천받는다
  ↓
한 문제를 푼 것이 아니라 한 개념을 배운다
```

이 경험의 핵심은 `문제 풀이 → 실행 이해 → 개념 설명 → 다음 문제`가 **하나의 루프로 끊기지 않고 이어진다**는 점이다.

> [!NOTE]
> CodeFlow는 정답 확인 도구가 아니라 **"실행 이해 + 개념 확장"을 만드는 학습 루프**다.

## 4. Learning Flow

### 4.1 주제를 선택한다

```
배열  /  투 포인터  /  재귀  /  DP
```

### 4.2 기본 문제로 시작한다

처음 문제는 **검증된 기본 문제(DB 기반)**에서 시작한다. 학습의 기준점을 만드는 역할이다.

| 역할 | 설명 |
|---|---|
| 대표 예시 제공 | 해당 주제의 전형적인 패턴 |
| 난이도 기준점 | AI 생성 문제의 레퍼런스 |
| Seed 역할 | AI 변형 문제의 원형 |
| 품질 안정성 | 신뢰할 수 있는 첫 경험 |

### 4.3 코드를 작성하고 실행한다

초기 언어 범위는 두 개로 제한한다.

[![Java](https://img.shields.io/badge/Java-ED8B00?style=flat-square&logo=openjdk&logoColor=white)](.)
[![Python](https://img.shields.io/badge/Python-3776AB?style=flat-square&logo=python&logoColor=white)](.)

> 언어를 많이 늘리는 것보다, **실행 과정 시각화가 잘 먹히는 학습 경험**을 먼저 만드는 것이 중요하다.

### 4.4 CodeFlow가 실행 과정을 시각화한다

- 현재 실행 라인
- Stack / Heap 상태
- Output
- 배열 / 포인터 이동
- 루프와 분기 변화

### 4.5 AI가 실행 과정을 설명한다

AI는 단순히 정답을 주는 존재가 아니다. **현재 실행 상태를 바탕으로 개념을 연결해 주는 튜터 역할**을 한다.

- 왜 여기서 `left`가 증가하는지
- 왜 이 시점에 `max`가 갱신되는지
- 현재 조건문이 왜 참인지
- 어디서부터 논리가 어긋났는지

### 4.6 같은 유형의 다음 문제를 제안한다

완전히 랜덤한 새 문제가 아니다. **"방금 푼 문제와 같은 유형이지만, 한 단계 더 배우게 만드는 문제"** 다.

| 이전 문제 | 다음 단계 |
|---|---|
| 배열 최댓값 | 최솟값 → 구간합 → 슬라이딩 윈도우 |
| 투 포인터 입문 | 정렬 + 분기 + 해 존재 여부 판단 |
| 피보나치 | 메모이제이션 → 점화식 응용 |

> AI는 단순 문제 생성기가 아니라 **다음 학습 단계를 설계하는 튜터**에 가깝다.

## 5. Main Entry Experience

메인 화면은 단순 랜딩이 아니라, 사용자가 **바로 학습을 시작할 수 있는 입구** 역할을 해야 한다.

### 5.1 AI에게 바로 요청하기

메인 화면에서 채팅창처럼 요청한다.

```
"투 포인터 입문 문제 만들어줘"
"배열 문제 하나 내줘"
"Python으로 재귀 문제 풀고 싶어"
```

AI는 요청을 해석해서 **주제 / 난이도 / 언어 / 학습 포인트**를 결정하고 맞춤 문제를 만든다.

### 5.2 카테고리 목록에서 고르기

모든 사용자가 채팅부터 시작하지는 않는다. 카테고리별 문제 목록을 탐색해서 직접 고를 수 있어야 한다.

```
배열  /  투 포인터  /  재귀  /  DP
```

### 5.3 중요한 포인트

채팅으로 만든 문제와 목록형 문제는 **분리된 세계가 아니다.**

채팅으로 생성된 문제도 검증을 통과하면 `AI Problem Library`에 쌓여야 한다. 그래야 메인 채팅이 단발성 기능이 아니라 **지속적으로 콘텐츠를 만드는 엔진**이 된다.

## 6. Product Differentiation

### 6.1 기본 문제에서 시작한다

처음부터 AI가 모든 문제를 만들면 품질이 흔들릴 수 있다. 시작은 **검증된 기본 문제 세트**로 간다.

- 사용자는 **안정적인 첫 경험**을 얻고
- 시스템은 **학습 기준점**을 확보하고
- AI는 그 기준점 위에서 **다음 문제를 생성**한다

### 6.2 다음 문제는 AI가 이어준다

CodeFlow의 재미는 "한 문제를 끝냈다"가 아니라  
**"지금 내 수준에 맞는 다음 문제로 이어진다"** 는 데 있다.

이 구조가 있어야 학습이 **선형이 아니라 누적형**이 된다.

## 7. Recommended Strategy

### DB + AI 하이브리드

완전 정적 문제집만 쓰면 문제풀이 사이트처럼 굳어지고, 완전 AI 생성만 쓰면 품질 리스크가 커진다. 가장 현실적인 구조는 **하이브리드**다.

```
기본 문제는 DB에서 제공
  ↓
사용자의 풀이 결과 / 약점 / 주제를 수집
  ↓
AI가 같은 유형의 변형 또는 다음 단계 문제 생성
  ↓
검증 통과 시 출제
  ↓
실패 시 DB fallback 문제 제공
```

| 역할 | 담당 |
|---|---|
| 🛡️ **안전망** | DB 기본 문제 |
| 🤖 **개인화 엔진** | AI 생성 문제 |

## 8. AI Problem Library

AI가 만든 문제는 **한 번 출제되고 사라지는 일회성 데이터가 아니라, 계속 쌓이는 학습 자산**이 되어야 한다.

### 왜 중요한가

- AI가 만든 문제를 **다시 풀 수 있다**
- 문제 생성 결과가 제품 안에 **콘텐츠로 축적**된다
- 같은 주제의 다양한 변형 문제를 **탐색**할 수 있다
- 추천 문제 / 복습 문제 / 인기 문제로 **확장**하기 좋다

### 탐색 기준

| 기준 | 예시 |
|---|---|
| **주제** | 배열 / 투 포인터 / 재귀 / DP |
| **난이도** | 입문 / 기본 / 응용 |
| **언어** | Java / Python |
| **생성 출처** | 기본 문제 / AI 변형 / 추천 문제 |
| **학습 포인트** | 경계값 / 정렬 / 슬라이딩 윈도우 / 포인터 이동 |

### 학습 아카이브처럼

겉으로는 목록 화면이어도, 실제로는 **학습 맥락이 살아 있는 구조**가 더 CodeFlow답다.

- 최근 생성된 문제
- 내가 저장한 문제
- 내가 틀렸던 유형과 비슷한 문제
- 지금 배우는 주제와 연결된 문제

### 페이지 흐름

```
Study
  └─ 기본 문제 풀이

Archive
  └─ AI가 생성한 문제 목록
      └─ 필터 (주제 / 난이도 / 학습포인트 / 언어)

Problem Detail
  ├─ 문제 설명
  ├─ 코드 작성
  ├─ 실행 시각화
  └─ AI 해설
```

## 9. Visualization Scope & Fallback Mode

CodeFlow는 모든 알고리즘 문제를 완벽하게 시각화하는 플랫폼이 아니다. **"우리가 잘 표현할 수 있는 범위를 깊게 보여준다"** 는 방향이 더 중요하다.

### Visualizer Mode (시각화 가능)

- ✅ 현재 실행 라인 시각화
- ✅ Stack / Heap / Output 표시
- ✅ 포인터 이동 / 루프 / 분기 흐름
- ✅ 단계별 AI 설명
- ✅ 다음 문제 추천

### Explain + Test Mode (시각화가 어려운 경우)

무리하게 시각화하려 하지 않고 **설명 중심 모드로 전환**한다.

- ✅ 문제 설명 및 테스트케이스 실행
- ✅ 정답 / 오답 판정
- ✅ AI 해설 및 접근 설명
- ✅ 시각화 가능한 하위 문제로 변형 제안

```
시각화 가능하면  →  깊게 시각화
시각화가 어렵다면  →  설명 + 테스트 중심으로 전환
```

> [!NOTE]
> "모든 문제를 시각화한다"보다 **"시각화 가능한 문제는 깊게, 아닌 문제도 학습은 이어준다"** 가 더 신뢰를 준다.

## 10. AI Responsibilities

AI는 최소 **4가지 역할**을 맡는다.

### 🔵 다음 문제 생성

- 현재 주제 유지
- 난이도 한 단계 조정
- 이전 문제와 과도하게 겹치지 않게 변형
- 학습 포인트가 분명해야 함

### 🔵 실행 과정 설명

- 현재 라인에서 무슨 일이 일어나는지
- 변수 상태가 왜 바뀌는지
- Stack / Heap이 무엇을 의미하는지
- 틀렸다면 어느 시점에서 어긋났는지

### 🔵 단계형 힌트 제공

바로 정답을 주지 않고 아래 순서로 제공한다.

```
1단계  개념 힌트
2단계  접근 힌트
3단계  코드 방향 힌트
4단계  정답 해설 (마지막에만)
```

### 🔵 학습 경로 추천

- 이 주제를 더 연습해야 하는지
- 다음에 어떤 유형으로 넘어가야 하는지
- 현재 부족한 개념이 무엇인지

## 11. Safety Layer

AI 생성 문제는 바로 출제하지 않는다. **문제 생성 뒤에는 반드시 검증 단계가 들어간다.**

### 검증 포인트

| 항목 | 기준 |
|---|---|
| 중복 여부 | 기존 문제와 지나치게 겹치지 않는가 |
| 내부 일관성 | 문제 설명과 제약이 서로 모순되지 않는가 |
| 예제 정확성 | 입력/출력이 올바른가 |
| 모호성 | 정답이 여러 개일 때 설명이 명확한가 |
| 실행 검증 | 생성된 정답 코드가 테스트를 실제로 통과하는가 |

### 운영 원칙

- 검증 실패 → **자동 재생성**
- 반복 실패 → **DB fallback 문제 제공**
- 품질 점수 미달 → **사용자에게 노출하지 않음**

> [!TIP]
> AI 생성은 **"무한 생성"보다 "검증된 생성"이 더 중요하다.**  
> 기능보다 품질 시스템이 먼저 설계되어야 한다.

## 13. MVP

초기 MVP는 아래 범위로 충분하다.

```
1  주제 선택
2  메인 화면에서 AI 채팅 또는 카테고리 탐색
3  기본 문제 제공
4  Java / Python 코드 작성
5  실행 결과 시각화  또는  Explain + Test 모드
6  AI 설명
7  다음 문제 AI 추천
8  AI Problem Library 누적
```

> 이 단계만 완성돼도 **CodeFlow의 핵심 가치는 분명해진다.**
